Why is header compression necessary in 6LoWPAN?
  • IPv6 mandates a minimum MTU of 1280 bytes, but IEEE 802.15.4 leaves only about 80–100 bytes of payload after frame and protocol overhead.
  • Without compression, IPv6 headers (~40 bytes) + UDP header (~8 bytes) would consume most of the available payload.
  • 6LoWPAN compresses IPv6/UDP headers to reduce per-packet overhead, avoid unnecessary fragmentation, and improve:
    • Payload efficiency
    • Energy consumption (fewer bytes to transmit)
    • Throughput on constrained links

What is LOWPAN_IPHC, and what benefit does it provide in 6LoWPAN?
  • LOWPAN_IPHC is the improved IPv6 header compression encoding defined in RFC 6282.
  • It compresses fields by elision or context-based inference:
    • Traffic Class / Flow Label (often all zeros → elided)
    • Hop Limit (common values → 1 byte encoding)
    • Source / Destination addresses (can be derived from MAC address or multicast prefix → fully elided)
  • Best case: IPv6 header compressed to as few as 2 octets.
  • Multi-hop / routing case: Even then, typically reduced to ~7 octets.
  • Combined with NHC (Next Header Compression), UDP header can be further compressed to ~4 bytes (from 8 bytes).
